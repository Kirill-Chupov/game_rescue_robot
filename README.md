# Robot Rescuer — изометрическая игра на C++/Qt

**Robot Rescuer** — это изометрическая игра, в которой вы управляете роботом, спасаете жертв в горящем здании, собираете ключи, тушите пожары и исследуете многоэтажные уровни. Проект разрабатывался в рамках учебного курса, состоящего из 10 этапов, и считается одним из самых сложных заданий, т.к. модуль был в тестовом режиме, который сняли на доработку.

Я не только выполнил все этапы, но и восстановил недостающие части, реализовал недостающий функционал и довёл проект до полностью рабочего состояния. Этот репозиторий — результат той работы, готовый для изучения, доработки или вдохновения.

**Цель игры**: найти всех пострадавших (`Victim`) и, возможно, вывести их (механика спасения заложена, но может быть расширена). На пути встречаются запертые двери, огонь, лестницы и другие препятствия.

---

## Геймплей

- Вы — робот, который может поворачиваться и перемещаться по клеткам.
- Жертвы сначала бродят случайно (броуновское движение), но, увидев робота, начинают следовать за ним.
- Чтобы открыть дверь, нужен ключ подходящего цвета.
- Огонь можно потушить только водой (воду даёт гидрант).
- Лестницы перемещают между этажами, если подойти к ним с правильной стороны.

---

## Особенности

- **Изометрическая графика** — весь мир отрисовывается с помощью `QPainter` в изометрической проекции с учётом перспективы и этажности.
- **Интеллект жертв** — конечный автомат с двумя состояниями: `BROWNIAN` (случайное блуждание) и `FOLLOW` (преследование игрока).
- **Динамическое освещение** — фонарик робота (`FlashlightDarkener`) динамически затемняет области в зависимости от расстояния и направления взгляда.
- **Инвентарь и предметы** — ключи (различаются по цвету) и вода; предметы можно подбирать и использовать.
- **Интерактивное окружение**:
  - Двери (`Door`) — открываются ключом (становятся невидимыми и проходимыми).
  - Огонь (`Fire`) — требует воды из инвентаря.
  - Гидрант (`Hydrant`) — пополняет запас воды.
  - Лестницы (`Stairs`) — телепортируют на другой этаж при подходе с нужной стороны.
- **Загрузка уровней из JSON** — гибкая система описания карт с помощью фабрик (`Loader<T>`). Позволяет создавать многоэтажные уровни произвольной формы (примеры: `demo_level.json`, `main_level.json`).
- **Анимации и плавные перемещения** — все перемещения интерполируются с помощью `Transition`, анимации объектов (огонь, ходьба робота) реализованы покадрово.
- **Тестируемость** — код подготовлен для юнит-тестирования (макрос `PRAC_TESTS` переключает генератор случайных чисел и таймер в детерминированный режим).
- **Модульная архитектура** — чёткое разделение на игровые объекты, поле, контекст, контроллер и утилиты. Код легко расширять и поддерживать.

## Управление

| Клавиша          | Действие                                                                     |
|-------------------|------------------------------------------------------------------------------|
| `←` `→` `↑` `↓`  | Поворот и движение. При удержании клавиши робот идёт непрерывно.            |
| (ЛКМ)             | В текущей версии не используется (зарезервировано для будущих механик).    |

Взаимодействие с объектами (двери, ключи, огонь, гидранты) происходит автоматически при попытке пройти сквозь них или при столкновении.

## Сборка и запуск

### Требования
- **Qt** 6 (или 5) с модулями `Core`, `Gui`, `Widgets`
- Компилятор с поддержкой **C++20** (MSVC 2019/2022, GCC 10+, Clang 12+)
- **qmake** (входит в состав Qt)

### Сборка в Qt Creator
1. Откройте корневой файл `robot-10-final-task.pro` в Qt Creator.
2. Выберите комплект сборки (например, Desktop Qt 6.5.3 MSVC2019 64-bit).
3. Установите конфигурацию **Release**.
4. Запустите сборку (Ctrl+B).

### Запуск
После сборки исполняемый файл находится в папке сборки (например, `build/submission/release/submission.exe`).  
Для запуска вне Qt Creator может потребоваться скопировать необходимые DLL Qt. Воспользуйтесь утилитой `windeployqt`:

```bash
windeployqt submission.exe --release
```

После этого `submission.exe` можно запускать двойным кликом.

По умолчанию загружается уровень, сгенерированный в коде (`main.cpp`). Чтобы загрузить уровень из JSON, раскомментируйте соответствующие строки в `main.cpp` (см. комментарии).

## Архитектура проекта

Проект построен вокруг нескольких ключевых классов и контекстов, обеспечивающих гибкость и расширяемость.

### Основные компоненты

- **`Game`** — главный контейнер, хранит поле, объекты, игрока и глобальный контекст.
- **`GameContext`** — агрегат ссылок на глобальные сервисы:
  - `ObjectMap<Object>` — карта размещения объектов на клетках.
  - `Field` — игровое поле с этажами.
  - `RandomGen` — генератор случайных чисел (с детерминированным режимом для тестов).
  - `AssetLoader` — загрузка спрайтов и тайлов.
  - `Timer` — планирование событий.
  - `Inventory` — инвентарь игрока.
- **`Object`** — базовый класс всех сущностей на карте. Содержит позицию, видимость, ссылку на `GameContext`. Определяет виртуальные методы `Draw`, `Interact`, `CanCover`.
- **`Character`** — наследник `Object` для подвижных персонажей. Добавляет направление, анимации, переходы между клетками (`Transition`), методы проверки возможности движения (`CanGo`, `CanPassWall`).
- **`Player`** — управляемый робот. Переопределяет `IsActive()` (всегда `true`), реализует команду движения `GoCommand()` (проверка препятствий и запуск анимации).
- **`Victim`** — жертва с конечным автоматом (`BROWNIAN` / `FOLLOW`). При столкновении с роботом переключается в режим следования.
- **`Field`**, **`Floor`**, **`Cell`** — иерархия для хранения игрового поля. `Floor` содержит двумерный массив клеток, каждая клетка может содержать пол (`Tile`) и стены (`Wall`).
- **`Wall`**, **`Tile`** — элементы окружения. Реализуют отрисовку, проходимость и взаимодействие.
- **`Inventory`** — управление предметами (ключи, вода). Хранит `shared_ptr<InventoryObject>`, позволяет искать, добавлять и удалять предметы.
- **`LevelLoader`** — загрузка уровней из JSON с помощью шаблонного `Loader<T>`, который регистрирует фабрики для разных типов объектов.
- **`Controller`** — обработка ввода, связь с игрой и обновление таймера. Содержит колбэк для перерисовки.
- **`MainWindow`** — главное окно, отрисовка через `Painter`, обработка клавиш, запуск таймера обновления (40 мс).

### Поток игры
1. Загрузка уровня (из кода или JSON).
2. `MainWindow` запускает таймер, который каждые 40 мс вызывает `Controller::Tick()`.
3. `Controller` обновляет время в `Timer` и инициирует перерисовку.
4. При нажатии клавиши `Controller::OnMoveKey()` либо меняет направление игрока, либо вызывает `Player::GoCommand()`.
5. `Player::GoCommand()` проверяет:
   - возможность пройти сквозь стену (`CanPassWall`),
   - возможность пройти в клетку (`CanGo`),
   - при необходимости вызывает `Interact` у стены или объектов.
   Если путь свободен — запускает `DoTransition()` (анимация перемещения).
6. Жертвы самостоятельно планируют свои шаги через `Timer` (метод `NextStep()`).
7. `Painter` преобразует мировые координаты в изометрические, применяет затемнение и рисует все объекты.

## Детали реализации

### Освещение (`FlashlightDarkener`)
- Вычисляет степень затемнения для каждой точки на основе вектора от игрока и направления взгляда.
- Darkness = 0 (полностью светло) … 1 (полностью темно).
- Используется скалярное произведение для выделения области перед роботом.

### ИИ жертв (`Victim`)
- **BROWNIAN**: каждые несколько шагов выбирает случайное направление (кроме текущего), проверяет возможность прохода. Если шаг невозможен — ждёт и меняет направление.
- **FOLLOW**: пытается идти к последней известной позиции игрока (`GetPrevPosition()`). Если расстояние превышает 5 клеток — возвращается в BROWNIAN.
- Все действия планируются через `Timer` с колбэками, обеспечивая асинхронность.

### Загрузка уровней из JSON
- Используется шаблон `Loader<T>` со статическим словарём фабрик.
- Для каждого типа объекта регистрируется функция, создающая соответствующий C++ объект.
- Пример регистрации в `loaders.cpp`:
  ```cpp
  Loader<Object>::AddLoader("victim", [](const QJsonObject& json, GameContext& context, Floor* floor) {
      return std::make_shared<Victim>(context, pos, dir);
  });
  ```
- Поддерживаются регионы тайлов, стены с длиной, объекты с произвольными параметрами.

### Анимации и переходы
- **Покадровые анимации** — `AnimatedAsset` хранит вектор кадров. Для огня — 21 кадр, для ходьбы робота — по 21 кадру на каждое направление.
- **Перемещение** — шаблон `Transition<T>` (наследник `Animation`) интерполирует положение между клетками. По окончании вызывает колбэк.

### Инвентарь и предметы
- `InventoryObject` — базовый класс предметов. Может быть ключом (`AsKey()`) или водой (`IsWater()`).
- `InventoryKey` — хранит имя и цвет, перекрашивает иконку.
- `InventoryWater` — маркер воды.
- `Inventory` предоставляет методы `Store`, `Remove`, `FindItem` (с предикатом).

### Коллизии и проходимость
- `Object::CanCover()` определяет, может ли персонаж пройти сквозь объект (по умолчанию — только невидимые).
- `Wall::CanPass()` вызывается для стены (например, дверь проходима, если открыта).
- `Character::CanGo()` проверяет все объекты в целевой клетке: все должны вернуть `true` на `CanCover()`.

## Тестирование

Макрос `PRAC_TESTS` переключает:
- `RandomGen` — возвращает заранее заданные значения (можно установить через `SetInRange`).
- `Timer` и `Clock` — позволяют вручную управлять временем (`SetCurrentTime`, `SetElapsedTime`).

Это позволяет писать детерминированные юнит-тесты. Тесты находятся в отдельной папке `tests/` (структура сохранена, но сами тесты не включены в данный репозиторий).

## Заключение
Проект родился на ранних этапах моего изучения C++ и стал серьёзной проверкой навыков. Разработка велась в рамках учебного курса, состоящего из 10 этапов. Начиная с 6-го этапа авторы из-за технической ошибки перестали предоставлять файлы, обеспечивающие единство интерфейса и сигнатур между проектом и тестовой системой (студентам не разрешалось их менять). Дальнейшее продвижение требовало самостоятельного восстановления недостающих частей и реализации функционала. Лишь немногие участники смогли дойти до финала - проект оказался сложнее, чем предполагалось.

Несмотря на множество ошибок, багов и моментов разочарования, мне удалось довести его до конца: все тесты пройдены, игра работает. Этот опыт дал мне понимание архитектуры игровых приложений, работы с графикой и асинхронного программирования.

В текущем виде игра представляет собой законченное ядро с базовыми механиками. Я не планирую активно развивать его дальше - создание полноценных уровней и полировка потребовали бы значительных усилий, а на тот момент я уже "выжал" из проекта максимум в учебных целях. Тем не менее, репозиторий может быть полезен как пример архитектуры игры на C++/Qt и отправная точка для тех, кто хочет создать нечто подобное.
